El llenguatge HTML

Introducció

A finals dels anys 80 es va desenvolupar el llenguatge de marques SGML. Al final d'aquesta dècada Tim Bernes Lee, científic britànic que treballava al CERN, va utilitzar SGML per definir un nou llenguatge d'etiquetes que va anomenar Hypertext Markup Language (llenguatge de marcat d'hipertext) per crear documents transportables a través d'Internet on fos possible l'hipertext.

S'anomena hipertext a la possibilitat que en un document determinades paraules marcades de manera especial permetin obrir un document relacionat amb elles; alguna cosa a què ara estem molt acostumats, però que abans era molt innovador.

A més d'HTML, l'equip de Tim Bernes Lee va definir les bases del protocol http d'Internet, encarregat de transportar els documents HTML i els recursos associats. La versió definitiva d'http es va fer l'any 1999 per part de l'organisme World Wide Web Consortium (W3C).

Tot i tardar a ser acceptat, HTML va ser un èxit rotund i la causa indubtable de l'èxit dInternet. Avui dia, quasi tot a Internet es veu a través de documents HTML que, popularment, s'anomenen pàgines web.

Inicialment aquests documents es veien amb ajuda d'intèrprets de text (com l'antigament famós Lynx d'Unix) que reconeixien el codi HTML però que tenien capacitat només de pintar el text especial (com les negretes i cursives per exemple) i remarcaven l'hipertext, sobre el qual la tecla tabulador permetia accedir de forma seqüencial a tots els hipervincles del document.

Després el programari va millorar i van aparèixer ja els anomenats navegadors amb capacitat d'interpretar el codi HTML de manera més visual i canviar el tipus de lletra, colors, incorporar imatges, so, etc. És a dir, realment els documents HTML es van convertir en pàgines web.

Lògicament des del 1989 fins als nostres dies HTML ha millorat. Entre els seus avanços fonamentals:

  • El llenguatge cada cop ha anat incorporant noves etiquetes més potents, que permeten incloure als documents HTML, taules, capes, marcs, imatges,…
  • S'han afegit llenguatges de script (especialment JavaScript) amb codi incrustat a les pròpies pàgines HTML que permeten afegir funcionalitats extra i dinamisme a les pàgines web
  • Has aparegut llenguatges i tecnologies que permeten executar accions avançades al servidor que allotja les pàgines web. Entre elles CGI, PHP, ASP o JSP.
  • S'han incorporat llenguatges d'estil (com CSS) per a generar un format de document més avançat i fàcil de mantenir.
  • S'han afegit utilitats per a gestió avançada de JavaScript amb XML (AJAX) per donar encara més interactivitat i dinamisme a les pàgines.
  • S'ha permès la inclusió d'elements avançats a les pàgines com Flash (obsolet) o els applets de Java per donar més funcionalitat.
  • Es permeten elements semàntics per donar significat al contingut.
  • S'afegeixen cada cop més plugins als navegadors que així són capaços de mostrar imatges, so, vídeo i altres elements multimèdia a les pròpies pàgines.
  • Els llenguatges relacionats amb les pàgines web segueixen creixent i possibiliten afegir-ho tot tipus d'elements a les pàgines web, dibuixos, animacions, ús de localització mitjançant GPS,…

Història de HTML

Es ressalten a continuació alguns dels esdeveniments més importants a la història d'HTML. S'indica l'any i el que va passar

  • 1989.
    • Tim Bernes Lee, científic britànic que treballava al CERN centre de desenvolupament nuclear ubicat a Suïssa. Intenta traslladar l'hipertext als documents científics, mitjançant el qual és possible avançar d'un document a un altre mitjançant enllaços existents al propi text. Va teoritzar la manera de transportar aquest tipus de documents (el actual protocol http) i sobre el llenguatge de marques a utilitzar.
  • 1990
    • Apareix el primer navegador web anomenat WorldWideWeb (més tard Nexus) per realitzar proves creat a un ordinador Next.
  • 1991
    • Tim Bernes Lee acudeix a un grup de discussió a Internet per discutir sobre com implementar l'hipertext de manera més convenient. Amb això no pretén privatitzar el seu invent sinó fer-ho públic des del primer moment.
  • 1992
    • Pei-Yuan Wei crea ViolaWWW, considerat el veritable primer navegador, usava un llenguatge de script a més d'HTML, era doncs molt innovador.
  • 1993.
    • El CERN anuncia que la web serà lliure per a tots, renuncia als possibles patents.
    • La NCSA s'interessa per la ja anomenada web de Bernes Lee i crea el primer navegador realment reeixit: Mosaic. Entre els seus creadors està Eric Bina i Marc Adreessen i va participar la universitat d'Urbana-Champaign futurs milionaris gràcies a la web.
    • Lou Montulli desenvolupa Lynx per als sistemes Unix, el primer navegador de text a la web. Serà àmpliament utilitzat en els anys següents, encara que després quedarà ràpidament superat per les capacitats dels navegadors gràfics
    • Es comença a fer popular l'etiqueta img (gràcies a Mosaic), les imatges comencen a poblar la web.
    • Apareix el primer esborrany d'HTML (Tim Bernes-Lee i Dan Conely)
  • 1994
    • Conferència global sobre la web.
    • La IETF assigna un grup de treball per estandarditzar HTML.
    • El llenguatge HTML comença a ser caòtic perquè apareixen nombroses etiquetes posades per cada entitat privada. Dan Connolly recopila les etiquetes HTML de l'època més utilitzada i es crea l'esborrany HTML 2.
    • Marc Adreessen i Jim Clark abandona la NCSA i funden Mosaic Communications (futura Netscape). Deixen també els estàndards i creen elements nous en el llenguatge HTML per crear pàgines més vistoses per al seu navegador.
    • A finales del año se crea la World Wide Consortium (W3C) fichando a algunos de los principales impulsores de la web (incluido Tim Bernes-Lee). Se convertirá en el principal organismo de estandarización de las tecnologías relacionadas con la web en general y de HTML en particular
    • L'empresa Mosaic Communications es converteix a Netscape Communications i llança el navegador Netscape Navigator. Es convertirà en els següents anys al navegador més utilitzat.
    • En diciembre aparece la versión 1.0 del Netscape Navigator. En poco tiempo pasa a ser el navegador más popular.
    • A finales de año hay 10000 servidores web
  • 1995
    • Continuen apareixent nous elements en HTML que impulsen les possibilitats de les pàgines web. Es crea l'esborrany HTML 3, que inclou tipus de lletra i altres millores.
    • Microsoft crea Internet Explorer i ho incorpora ràpidament com a part del sistema operatiu Windows 95. Comença la primera guerra dels navegadors. Els contendents són Explorer i Navigator.
    • El grup de treball de la IETF per a HTML es desmantella per la seva escassa influència. El World Wide Consortium queda com a principal organisme de estandardització d'HTML.
    • El navegador Opera es comença a comercialitzar. El desenvolupa Telenor, una empresa noruega.
    • A finals d'any apareixen els primers elements de creació de fulls d'estil, arrel del llenguatge CSS que permet donar format avançat a les pàgines web i que segueix sent una de les tecnologies imprescindibles actualment per crear pàgines web.
    • Sun Microsystems crea el llenguatge Java, que tindrà una enorme influència en el desenvolupament de Internet.
    • Els germans Allaire, creen ColdFussion, un llenguatge basat en HTML que s'executa al servidor que allotja les pàgines web (servidors compatibles amb aquesta tecnologia) de manera que el client no necessita tenir un programari especial que reconegui aquesta tecnologia. Al client li arriben pàgines web normals que ha preparat el servidor després de traduir aquest llenguatge.

    Va ser la primera tecnologia de script al costat del servidor.

    • Netscape desarrolla JavaScript, un llenguatge basat en C i Java que s'incrusta dins del codi HTML de les pàgines per donar-los una major potència. Encara continua sent un dels llenguatges més influents en el desenvolupament de pàgines i aplicacions per a la web. Netscape Navigator 2.0 és el primer navegador a utilitzar aquest llenguatge

    En definitiva és una tecnologia de script al costat del client. Els navegadors, a causa del seu èxit, van haver d'anar a poc a poc incorporant plugins (programari afegit) que permetessin executar codi JavaScript.

  • 1997

  • 1998

Funcionament de les aplicacions web

HTML, CSS i JavaScript

Una aplicació web, és una aplicació creada usant com a base el llenguatge HTML. Per tant, és una aplicació que s'executa en un navegador d'Internet.

Les aplicacions web utilitzen llenguatges que donen suport a la tasca d'HTML i que són traduïbles per part dels navegadors. Essencialment les aplicacions web actuals utilitzen:

  • HTML. Per donar significat als continguts de laplicació web. Permet indicar quins textos són títols, quins són paràgrafs normals, quines són cel·les duna taula, quines són imatges, etc.
  • CSS. Llenguatge que permet donar format i maquetació als continguts. Color, mida de lletra, posició, etc.
  • JavaScript. Permet dissenyar la interactivitat de la pàgina. Permet que les accions de l'usuari es puguin capturar i que la pàgina hi reaccione. Exemples de les possibilitats que quan l'usuari agafi el ratolí a una imatge aquesta s'il·lumine, o que quan l'usuari faci clic en un títol es mostre el contingut relacionat si estava ocult o que l'usuari puga augmentar la mida de la lletra de la pàgina fent clic a una icona.

Protocol http

La transmissió de pàgines web (que en definitiva són documents HTML) es fa mitjançant el protocol http, que és part de la pila de protocols TCP/IP. És un protocol basat en una comunicació petició-resposta; de manera que un client (també anomenat user agent, agent d'usuari) fa una petició de recurs indicant la seva adreça a Internet. La petició arribarà a un servidor http (també anomenat servidor web), el qual respon a aquesta petició, bé transmetent al client el recurs sol·licitat o bé indicant un missatge derror si el recurs no està disponible.


Els paquets http que s'intercanvien entre el client i el servidor contenen una capçalera amb informació de control i després el cos. Aquest cos és el codi HTML en el cas duna pàgina web, o la imatge, àudio, etc. que s'està transmetent.


A l'imatge anterior, s'observa el paquet retornat pel servidor web quan rep una petició http (de tipus GET) a la URL http://jorgesanchez.net. S'indica que la resposta va ser correcta (codi 200, OK), la IP del servidor que respon, la mida del paquet, la data en què expira, etc. A més s'annexa la capçalera de la petició original (request headers) on es veu el navegador de l'usuari que va fer la petició, el llenguatge del seu equip, etc.

A més de la capçalera, el contingut dels paquets http té un cos amb el contingut del recurs que se sol·licitava. Per exemple, en cas d'haver sol·licitat una pàgina web, el cos conté codi HTML que el navegador podrà traduir.

El protocol http serveix per transportar tota mena de continguts. De fet, el codi HTML d'una pàgina web fa referència a altres fitxers (imatges, fitxers CSS, etc.). Quan el navegador detecta aquestes referències, sol·licita al servidor web aquests recursos. El navegador respondrà amb nous paquets que contindran (si està a la mà) el contingut sol·licitat.

El client (normalment un navegador) serà l'encarregat de mostrar el recurs sempre que el navegador tinga la capacitat de fer-ho, ja que els navegadors poden mostrar molts formats de contingut, però per descomptat no tots.

Entre els recursos transmesos per http que un navegador és capaç de reconèixer estan:

  • Codi HTML, CSS i JavaScript
  • Imatges JPEG, PNG, SVG i GIF
  • Vídeos MP4 i altres formats
  • Àudio MP3 i altres formats
  • Fitxers XML (encara que no la validació ni el codi XSL associat)

Front-end i back-end

En aquest apartat s'explica breument el funcionament d'una aplicació web complexa. L'únic propòsit és indicar que les aplicacions web utilitzen nombroses tecnologies.

A la majoria d'aplicacions web complexes, quan un navegador requereix aquesta aplicació, el servidor necessita executar accions com: interpretar codi del costat del servidor (per exemple en PHP o ASP.Net), demanar recursos a altres servidors (bases de dades, mapes, streaming de vídeo, etc.) o emmagatzemar dades de sessió.

Aquestes accions són totalment opaques a l'usuari. Després de la seva execució, el navegador de l'usuari rep codi que sí que és capaç d'interpretar (normalment HTML, CSS i JavaScript).

Això implica que una aplicació web pot realitzar tasques a l'anomenat costat del servidor. És a dir, tasques que s'executen al servidor i el codi del qual no podem obtenir. Per tant, quan es crea una aplicació web hi ha persones dedicades a programar en codi traduïble al costat del servidor i codi al costat del client,

El costat del servidor s'associa al desenvolupament back-end, els seus programadors són anomenats programadors back-end. Mentre que els professionals encarregats de l'aparença al navegador, són anomenats programadors front-end.


HTML és la tecnologia dominant per programar el front-end. Per tant, aquest és un "manual" dedicat al front-end.

Eines per crear aplicacions en HTML

Per escriure codi HTML n'hi hauria prou amb un editor de text pla com el bloc de notes de Windows o vi veure de Linux/Unix. No obstant això, l'escriptura d'HTML en aquest tipus d'editors és incòmoda ja que no proporcionen ajuda ni acceleren l'escriptura.

A més, els resultats s'han de mostrar en un navegador o, encara millor, provar en diversos navegadors per comprovar problemes de compatibilitat (ja que hi ha elements HTML que no són compatibles amb tots els navegadors).

Així, la llista de navegadors a instal·lar per provar el nostre codi seria: Microsoft Internet Explorer, Google Chrome, Mozilla Firefox, Apple Safari i Opera. D'altra banda, és interessant provar en dispositius mòbils o bé afegir plugins als nostres navegadors que simulin la visualització d'un mòbil.

Per escriure el codi l'ideal és treballar mitjançant editors de codi capaços d'entendre el llenguatge i pintar de manera diferent les etiquetes HTML per distingir-les del text normal i així treballar millor i que a més incorporin eines que facilitin l'edició.

Així tenim les opcions següents:

  • Editors de text multipropòsit (editors de codi). Són programes que permeten editar text, però que estan pensats per escriure codi a qualsevol llenguatge de programació. El seu avantatge és que acoloreixen la sintaxi depenent del llenguatge en què estem escrivint, perquè són editors políglotes.
    Quan escrivim en HTML, acoloreixen el text ressaltant el codi de manera que sigui més llegible. A més ens ajuden a escriure codi HTML de forma més eficaç, perquè fins i tot detecten les fallades. Entre els més coneguts hi ha Notepad++, TextMate, Atom, Visual Studio Code o Sublim Text.
  • Editors de text HTML. Són editors especialitzats a escriure codi HTML. La diferència respecte als anteriors és que, amb aquests, no podrem escriure codi en llenguatges com C o Java. A canvi, tindrem més funcions per escriure en HTML. Per exemple, podrem previsualitzar el resultat del codi HTML (avançant-nos al navegador), pintat més ajustat als llenguatges de la web, reconeixement de llibreries i plantilles d'ús habitual de dissenyadors,…
    Alguns són: Coda, Komodo Edit, WebStorm o Brackets. Els editors Atom i Visual Studio Code n'hi ha que els col·loquen en aquest tipus perquè, encara que són editors políglotes, estan molt especialitzats en aplicacions web.
  • Editors XML. Pensats per a la gent que treballa creant documents i elements relacionats amb XML. Entenen HTML, però no són tan potents com els anteriors, encara que poden servir per a feines que no requereixen molta especialització en HTML. Els editors XML més famosos són Oxygen i XMLSpy.
  • Entorns complets de programació. Són els anomenats IDE (Entorns de Desenvolupament Integrat) que són entorns de treball per programar en llenguatges avançats com Java o C++, però que solen treballar perfectament per crear projectes per a la web. Destaquen Eclipse, Aptana (versió d'Eclipse per a llenguatges a la web), NetBeans, IntelliJ IDEA i Microsoft Visual Studio.
  • Editors en línia. Són aplicacions web que ens permeten escriure i provar el nostre HTML. Per exemple, l'editor de w3schools:
    http://www.w3schools.com/html/tryit.asp?filename=tryhtml_basic
    o bien
    http://htmledit.squarefree.com/
    Per a petits fragments es fa servir https://codepen.io
  • Editors visuals. Permeten treballar de manera que puguem construir pàgines web sense escriure codi, de manera visual. Es basen en el programari WYSIWYG (What You See Is What You Get), el que es veuen a la pantalla és el resultat que s'obtindrà. Referit al fet que treballes veient la feina tal com quedarà finalment. En realitat és impossible ja que l'aspecte final dependrà del dispositiu i el programari que s'utilitzi en veure la pàgina i això variarà enormement d'un usuari a un altre; però com a mínim permeten fer-se una idea molt visual del resultat.
    Adobe Dreamweaver seria el més famós d'aquest tipus i el més complet, ja que també permet treballar còmodament amb el codi. Altres opcions són Adobe Muse, Tumult Hype, Google Web Designer, Kompozer, WYSIWYG Web Builder o Xara Web Designer.

Versions d'HTML

Els estàndards que segueixen en funcionament actualment són els següents:

HTML 4.01

Es tracta de la versió estàndard de l'HTML tradicional feta l'any 1999 i que segueix tenint certa vigència actualment, ja que és l'últim HTML l'estàndard del qual va definir la W3C abans de XHTML.

Per avisar que el nostre document segueix les normes d'HTML 4.01, es col·loca a les pàgines web, una etiqueta DOCTYPE (ha de ser la primera línia del document) que serveix per avisar precisament de quin tipus d'HTML segueix el document i quin document oficial li valida. Per a HTML 4.01 transicional (hi ha una versió d'HTML4.01 estricta i una altra per a les, ja en desús, pàgines de marcs) és:

<!DOCTYPE HTML PUBLIC
  "-//W3C//DTD HTML 4.01 Transitional//EN"
  "http://www.w3.org/TR/html4/loose.dtd">


XHTML

S'ideà com el substitut d'HTML, ja que es considerava que XML quedaria com a llenguatge de marques definitiu i que calia compatibilitzar HTML amb les normes d'XML. La primera versió, la 1.0 segueix sent la més utilitzada. Hi ha també una versió estricta, una transicional i una versió per a pàgines amb marcs (que gairebé no es fa servir). La més utilitzada és la transicional (igual que a HTML 4) perquè permet l'ús de nombrosos atributs i elements que es consideren obsolets, però que molts dissenyadors utilitzen.

Les pàgines XHTML obliguen a que l'escriptura d'HTML segueixi les regles del llenguatge XML ben format, per la qual cosa són més estrictes en la manera d'escriure.

Per indicar que utilitzem XHTML 1.0 transicional cal posar aquesta etiqueta DOCTYPE al principi del document:

<!DOCTYPE html PUBLIC
"-//W3C//DTD XHTML 1.0 Transitional//EN"
"http://www.w3.org/TR/xhtml1/DTD/xhtml1-transitional.dtd">


A més, l'etiqueta arrel html de tot document HTML s'ha d'usar d'aquesta manera (especificant l'espai de noms i l'idioma de la pàgina):

<html xmlns="http://www.w3.org/1999/xhtml"xml:lang="ca">


En el cas de XHTML 1.0 estricte, el DOCTYPE seria:

<!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Strict//EN" "http://www.w3.org/TR/xhtml1/DTD/xhtml1-strict.dtd">


L'etiqueta arrel html seria igual que a la versió transicional

HTML 5

Es considera el estàndard actual. A poc a poc s'està convertint en HTML més utilitzat. Es va crear fora del World Wide Web Consortium a l'entitat anomenada WHATWG (Web Hypertext Application Technology Working Group), afavorida per Apple, Google i Mozilla entre altres empreses a les quals s'han anat afegint moltes més.


La W3C va decidir unir-se a la WHATWG el 2008 per buscar un estàndard. Actualment HTML 5 ja és una recomanació oficial de la W3C que ha substituït WHATWG com a organisme encarregat de vetllar per l'estàndard.

HTML5 segueix una filosofia d'“estàndard viu” en què ja hi ha una recomanació d'ús per part seva, però que va canviant contínuament segons es consideren noves millores o l'aparició de noves tecnologies; aquesta forma dinàmica de fer servir les recomanacions ha estat la clau de l'èxit d'HTML 5 des de l'any 2012 (especialment gràcies que els dispositius Apple i Android sempre han manejat molt bé HTML 5).

Des del 2014 és la recomanació oficial de la W3C.

HTML 5 segueix creixent i per això (seguint la filosofia d'estàndard dinàmic de la WHATWG) les pàgines HTML 5 indiquen que usen aquesta versió d'HTML mitjançant un DOCTYPE molt més senzill: 

<!DOCTYPE html>
<html lang="es">


L'etiqueta arrel HTML també és més senzilla que les utilitzades en XHTML (indicant simplement l'idioma).

La versión actual és l'estàndard HTML 5.3

Validar el cod HTML

Els navegadors no són estrictes amb les normes HTML; és a dir, encara que tinguem algunes fallades, els ignoraran per mostrar la millor versió del codi que fem. Als navegadors els interessa fer funcionar el codi, sigui vàlid o no.

Tot i això, com a creadors de pàgines web, hem de complir les normes del llenguatge aprenent així realment el llenguatge ia més impedint que el navegador s'equivoqui en interpretar les nostres fallades. A més, fa que altres creadors de pàgines comprenguin el codi perfectament, facilitant el treball col·laboratiu.

El validador oficial de la W3C (http://validator.w3.org/) permet validar totes les versions HTML, inclosa la versió 5.

Els errors greus es mostren en vermell i hi pot haver avisos (warnings) que són errors lleus, però que cal tenir en compte.

D'altra banda, la major part dels entorns de treball disposa de validador incorporat (al mateix entorn es marquen els errors alhora que escrivim el codi) i alguns altres es pot instal·lar un plugin.

També podem instal·lar validadors en línia d'ordres com l'htmltidy, disponible a http://www.html-tidy.org/, que permeten validar des de la línia d'ordres, encara que aquesta opció cada cop es fa servir menys.

Compatibilitat entre navegadors

La tecnologia dominant actualment per crear pàgines web és la que sorgeix dHTML5 i que inclou lús del llenguatge CSS (versió 3) i de JavaScript.

Tot i que avui dia gairebé tots els navegadors suporten HTML5, molts (sobretot si no estan actualitzats) no suporten totes les característiques. De fet, com que HTML5 és viu i segueix introduint noves possibilitats, no hi ha cap navegador que inclogui totes les opcions.

Tradicionalment, Internet Explorer no és un navegador respectuós amb HTML 5, de fet només a partir de la versió 9 suporta els elements fonamentals d'HTML5 i només a partir de la 10 ho fa de manera realment comparable a la de la resta de navegadors actuals. De fet, o es recomana als usuaris actuals fer servir aquest navegador, excepte ara mostrar aplicacions web realitzades fa anys i que només són compatibles amb aquest, antany dominant, navegador.

El nou navegador de Microsoft, Edge sí que és més compatible. Els navegadors en, general Chrome i Firefox són els que adopten més ràpid les noves normes i aporten més compatibilitat amb HTML5, però depèn de propietats. Altres navegadors com Safari i Opera reconeixen també la immensa majoria d'HTML5.

La pàgina caniuse.com és la referència més completa i utilitzada per saber quines característiques dels llenguatges utilitzats a les pàgines web són suportades a cada navegador.


D'altra banda, el lloc http://html5test.com/ realitza un test al navegador amb què entrem, per determinar el grau en què suporta HTML 5 (li dóna una puntuació) i ens indica quines característiques compleix i quines no ( cal tenir en compte que la tecnologia HTML5 no només es refereix al propi HTML sinó al CSS i JavaScript relacionat, així com a altres tecnologies dús amb HTML).

Una altra opció és http://fmbip.com/ que mostra les capacitats relacionades amb HTML5 que té el nostre navegador.

Una altra possibilitat interessant és utilitzar la llibreria Modernizr (disponible a http://modernizr.com/) que permet als desenvolupadors a JavaScript detectar les característiques que té el navegador de l'usuari i actuar en conseqüència des del codi mateix.

Davant de tants canvis i incompatibilitats, la situació és difícil per aprendre. Per això hi ha adreces a Internet que ens ajuden a utilitzar els diferents elements HTML5 (fins i tot ens en diuen la compatibilitat). Les més importants són:

  • w3schools. Conté informació al dia sobre HTML, CSS, JavaScript i altres tecnologies web. Està en anglès, però és senzilla i ben estructurada. L'ajuda de les etiquetes HTML és a l'adreça:
    http://www.w3schools.com/tags/
  • Mozilla Developer Network. La pàgina oficial dels desenvolupadors del navegador Mozilla Firefox està fins i tot millor organitzada que l'anterior. Conté informació de tots els elements HTML a mesura que apareixen, encara que no estiguen del tot estandarditzats, indicant-ne la compatibilitat amb els navegadors i si ja és estàndard o no. Això sí, fa més èmfasi en els elements que el navegador Mozilla Firefox és capaç de reproduir.
    https://developer.mozilla.org/en-US/docs/Web/HTML/Element
  • World Wide Web Consortium (W3C). Referència oficial de la documentació de tots els estàndards web. S'estructura de manera molt formal, però té l'avantatge que aquesta documentació és la que fa referència a l'estàndard. Sobre HTML podem acudir a l'adreça:
    https://www.w3.org/TR/html5/

Publicació de llocs web

S'anomena lloc web el conjunt de pàgines web i recursos que contenen tota la informació associada a una determinada adreça d'inici a Internet.


Quan una persona desitja crear un nou lloc web, inicialment us crea al vostre ordinador de treball i per això ha de crear una carpeta i en ella emmagatzemar totes les pàgines i recursos necessaris (imatges, sons, vídeos, arxius auxiliars,…). Aquesta carpeta s'haurà d'enviar al servidor web que hàgim contractat o del que disposem per publicar la nostra pàgina a Internet.

Per això normalment s'utilitza el protocol de transmissió de fitxers conegut com a FTP, encara que és possible que aquesta transmissió es faça amb altres protocols com WebDAV o RDS. Amb copiar la carpeta al lloc adequat del nostre servidor, la pàgina estarà publicada. Normalment per això se'ns demana un usuari i una contrasenya que verifica que realment som els propietaris de l'espai.

Una altra opció, més habitual avui dia per als desenvolupadors, és utilitzar sistemes de control de versions al núvol (el més conegut és GitHub) i descarregar l'última versió del codi directament al servidor web. És a dir, des del nostre ordinador pugem la darrera versió del nostre codi al servidor al núvol i després actualitzem el codi al servidor web.
